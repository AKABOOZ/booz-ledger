# CONVENTIONS_AND_DECISIONS

## 1. 代码规范

### 当前事实

- 当前项目并没有形成完善、强约束的工程规范文档。
- 实际代码风格以 Flutter 默认风格 + 项目既有写法为主。
- 由于主业务集中在单文件，很多规范更像“约定俗成”而不是被工具强制执行。

### 命名规范

- 类型名：
  - 使用 UpperCamelCase
  - 例如：`LedgerStore`、`LedgerEntry`、`SettingsPage`
- 私有类 / 私有方法 / 私有字段：
  - 使用前导下划线 `_`
  - 例如：`_LedgerAppState`、`_save()`、`_storageKey`
- 变量 / 方法：
  - 使用 lowerCamelCase
- 常量：
  - 文件内常量通常用 `const` + lowerCamelCase / `_privateCamelCase`
  - 当前并没有统一使用 `SCREAMING_SNAKE_CASE`
- 枚举：
  - 使用 UpperCamelCase 枚举名，lowerCamelCase 枚举值

### 注释规范

- 当前注释偏少，主要用于：
  - 解释功能块
  - 记录流程意图
  - 标记局部临时说明
- 建议后续继续保持“少而有用”的注释风格：
  - 解释为什么这样做
  - 不要重复代码字面意思

### 文件拆分规则

### 当前现实

- 暂时没有被执行的文件拆分规则。
- 当前主业务全部集中在 `lib/main.dart`。

### 建议接手规范

- 新功能若较小，可先延续现状，避免大规模搬迁带来不必要风险。
- 新功能若较大，建议按以下方向逐步拆分，而不是继续堆进 `main.dart`：
  - `models/`
  - `stores/` 或 `state/`
  - `services/`
  - `pages/`
  - `widgets/`
  - `sync/`
  - `import_export/`

### UI 约定

- 使用 Material 3
- 主色系偏青绿色
- 字体统一为 `NotoSansSC`
- 大量使用：
  - `showModalBottomSheet`
  - `MaterialPageRoute`
  - 自定义卡片式布局

## 2. 路由规范、命名规则

### 当前路由方式

- 当前项目没有引入命名路由库，也没有使用 `go_router` / `auto_route`。
- 主导航是 Tab Shell：
  - 首页容器在 `LedgerHome`
  - 主要 tab 为：
    - 流水明细
    - 统计分析
    - 账户管理
- 二级页面主要通过以下方式进入：
  - `Navigator.of(context).push(MaterialPageRoute(...))`
  - `showModalBottomSheet(...)`
  - `showDialog(...)`
  - 少量自定义 `PageRoute`

### 当前路由命名约定

- 页面类名一般以 `Page` 结尾
  - 例如：`SettingsPage`、`StatisticsPage`、`AccountFormPage`
- 底部弹层 / 选择器一般以 `Sheet`、`Picker`、`RoutePage` 命名
  - 例如：`_CategoryPickerSheetBody`
  - `showAccountPickerSheet`

### 后续建议

- 如果继续维持当前体量：
  - 继续使用 `MaterialPageRoute + BottomSheet` 即可
- 如果页面进一步增多：
  - 再考虑统一引入声明式路由
  - 但这是重构项，不是短期必要项

## 3. 之前做过的关键技术决策

以下内容来自当前代码现状与最近开发上下文推断，是接手时最应该知道的设计选择。

### 决策 1：主线继续用 Flutter，而不是根目录 React/Vite

- 仓库里虽然有 React/Vite 脚手架，但当前实际可运行、已持续维护的主应用是 Flutter
- 证据：
  - 真正业务入口是 `lib/main.dart`
  - Android/iOS 构建、安装和最近所有改动都围绕 Flutter 工程进行

### 决策 2：不做自建后端，优先单机化

- 当前没有服务器端应用
- 所有核心账本逻辑都在本地执行
- 同步和智能识别都直接走第三方服务

原因：

- 个人记账 App 先追求离线可用和开发速度
- 避免维护额外服务端

代价：

- 配置分散在客户端
- 数据结构演进和同步冲突治理能力较弱

### 决策 3：账本主数据先存在 SharedPreferences

- `LedgerStore` 把账户、流水、自定义分类整体序列化为 JSON 保存

原因：

- 实现简单
- 初期开发快
- 方便导入导出

代价：

- 数据量大时性能、稳定性、可维护性都不如数据库
- 是明确的技术债

### 决策 4：智能识别采用“百度 OCR / 语音 + 大模型理解”的串联模式

- 图片识别不是直接喂图给模型，而是：
  - 先走百度 OCR 提文本
  - 再让 DeepSeek / Qwen 根据 OCR 文本推断金额、分类、账户、备注
- 语音识别先转文本，再按规则 / AI 回填

原因：

- 降低直接多模态接入复杂度
- 保持账单字段映射可控

### 决策 5：同步能力选 WebDAV / NAS

- 当前同步目标不是项目私有云，而是用户自己的 NAS / WebDAV

原因：

- 对个人用户成本低
- 不需要自建服务
- 用户可掌握数据所有权

### 决策 6：分类体系以“固定大类 + 可追加小类”为主

- 代码里内置了较完整的大类和小类
- 自定义分类是附加到固定组下，而不是完全自由树形结构

原因：

- 统计更稳定
- AI / OCR 分类映射更容易做

代价：

- 灵活度有限
- 分类管理页面也明确提示当前版本不开放完整编辑

### 决策 7：本地备份只留 1 份，NAS 留 9 份

- 最近一次调整后：
  - 本地只保留最新 1 份备份
  - NAS 仍保留最新 9 份
  - WebDAV 同步过程中的本地中间备份会自动删掉

原因：

- Android 本地“用户数据”曾因备份累积明显膨胀
- 用户希望节省手机存储
- NAS 端仍保留更多历史作为兜底

### 决策 8：暂不重构单文件，优先继续交付功能

- 尽管 `lib/main.dart` 很大，但最近几次需求仍是在原结构上快速迭代

原因：

- 当前目标是功能交付速度
- 全量拆分重构风险高

代价：

- 新人 / 新 AI 接手理解成本高

## 4. Codex 开发时的对话要点总结（关键需求变更）

以下是最近一轮开发中，已经确认并落地的关键需求 / 事实，接手者应默认成立。

### 需求变更 1：高级筛选增加“取消全选”

- 页面：高级筛选
- 范围：
  - 分类
  - 账户
- 规则：
  - 都要增加“取消全选”按钮
  - 点击后分别一键取消所有分类 / 所有账户
  - 当内部选项没有全选时，不显示该按钮

### 需求变更 2：本地备份从“多份保留”调整为“只保留 1 份”

- 用户明确表示：手机本地没有必要保留 9 份备份，1 份即可
- 当前代码已经按此调整

### 需求变更 3：NAS 备份策略不改，仍保留 9 份

- 用户确认：
  - 本地保留 1 份即可
  - NAS 继续保留过去的 9 份历史

### 需求变更 4：排查并清理 Android 用户数据膨胀

- 已排查出历史问题：
  - 本地 `ledger_backup_*.json` 大量累积
- 已执行处理：
  - 清理设备上的冗余本地备份
  - 清理 cache 中的分享 / 选图残留
- 结论：
  - 用户数据下降明显
  - 仍保留的约百兆数据不全是垃圾，debug 运行资产也会占空间

### 需求变更 5：Android 安装流程的经验结论

- `flutter build apk --debug` 在这个项目里可能出现“假失败”
- 即使终端说失败，也要继续检查：
  - `android/app/build/outputs/flutter-apk/app-debug.apk`
  - `android/app/build/outputs/apk/debug/app-debug.apk`
- 最近多次安装都依赖这个经验完成

### 对后续 AI 的提醒

- 不要把根目录 React/Vite 工程当成主线
- 不要轻易回退备份清理逻辑
- 修改同步、导入导出、账户余额联动时，要一起验证数据一致性
- 在没有充分预算前，不要贸然做“大拆 main.dart”式重构
