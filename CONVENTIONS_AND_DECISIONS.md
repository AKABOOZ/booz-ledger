# CONVENTIONS_AND_DECISIONS

## 1. 代码规范

### 当前事实

- 当前项目并没有形成完善、强约束的工程规范文档。
- 实际代码风格以 Flutter 默认风格 + 项目既有写法为主。
- 由于主业务集中在单文件，很多规范更像“约定俗成”而不是被工具强制执行。

### 命名规范

- 类型名：
  - 使用 UpperCamelCase
  - 例如：`LedgerStore`、`LedgerEntry`、`SettingsPage`
- 私有类 / 私有方法 / 私有字段：
  - 使用前导下划线 `_`
  - 例如：`_LedgerAppState`、`_save()`、`_storageKey`
- 变量 / 方法：
  - 使用 lowerCamelCase
- 常量：
  - 文件内常量通常用 `const` + lowerCamelCase / `_privateCamelCase`
  - 当前并没有统一使用 `SCREAMING_SNAKE_CASE`
- 枚举：
  - 使用 UpperCamelCase 枚举名，lowerCamelCase 枚举值

### 注释规范

- 当前注释偏少，主要用于：
  - 解释功能块
  - 记录流程意图
  - 标记局部临时说明
- 建议后续继续保持“少而有用”的注释风格：
  - 解释为什么这样做
  - 不要重复代码字面意思

### 文件拆分规则

### 当前现实

- 暂时没有被执行的文件拆分规则。
- 当前主业务全部集中在 `lib/main.dart`。

### 建议接手规范

- 新功能若较小，可先延续现状，避免大规模搬迁带来不必要风险。
- 新功能若较大，建议按以下方向逐步拆分，而不是继续堆进 `main.dart`：
  - `models/`
  - `stores/` 或 `state/`
  - `services/`
  - `pages/`
  - `widgets/`
  - `sync/`
  - `import_export/`

### UI 约定

- 使用 Material 3
- 主色系偏青绿色
- 字体统一为 `NotoSansSC`
- 大量使用：
  - `showModalBottomSheet`
  - `MaterialPageRoute`
  - 自定义卡片式布局

## 2. 路由规范、命名规则

### 当前路由方式

- 当前项目没有引入命名路由库，也没有使用 `go_router` / `auto_route`。
- 主导航是 Tab Shell：
  - 首页容器在 `LedgerHome`
  - 主要 tab 为：
    - 流水明细
    - 统计分析
    - 账户管理
- 二级页面主要通过以下方式进入：
  - `Navigator.of(context).push(MaterialPageRoute(...))`
  - `showModalBottomSheet(...)`
  - `showDialog(...)`
  - 少量自定义 `PageRoute`

### 当前路由命名约定

- 页面类名一般以 `Page` 结尾
  - 例如：`SettingsPage`、`StatisticsPage`、`AccountFormPage`
- 底部弹层 / 选择器一般以 `Sheet`、`Picker`、`RoutePage` 命名
  - 例如：`_CategoryPickerSheetBody`
  - `showAccountPickerSheet`

### 后续建议

- 如果继续维持当前体量：
  - 继续使用 `MaterialPageRoute + BottomSheet` 即可
- 如果页面进一步增多：
  - 再考虑统一引入声明式路由
  - 但这是重构项，不是短期必要项

## 3. 之前做过的关键技术决策

以下内容来自当前代码现状与最近开发上下文推断，是接手时最应该知道的设计选择。

### 决策 1：主线继续用 Flutter，而不是根目录 React/Vite

- 仓库里虽然有 React/Vite 脚手架，但当前实际可运行、已持续维护的主应用是 Flutter
- 证据：
  - 真正业务入口是 `lib/main.dart`
  - Android/iOS 构建、安装和最近所有改动都围绕 Flutter 工程进行

### 决策 2：不做自建后端，优先单机化

- 当前没有服务器端应用
- 所有核心账本逻辑都在本地执行
- 同步和智能识别都直接走第三方服务

原因：

- 个人记账 App 先追求离线可用和开发速度
- 避免维护额外服务端

代价：

- 配置分散在客户端
- 数据结构演进和同步冲突治理能力较弱

### 决策 3：账本主数据先存在 SharedPreferences

- `LedgerStore` 把账户、流水、自定义分类整体序列化为 JSON 保存

原因：

- 实现简单
- 初期开发快
- 方便导入导出

代价：

- 数据量大时性能、稳定性、可维护性都不如数据库
- 是明确的技术债

### 决策 4：智能识别采用“百度 OCR / 语音 + 大模型理解”的串联模式

- 图片识别不是直接喂图给模型，而是：
  - 先走百度 OCR 提文本
  - 再让 DeepSeek / Qwen 根据 OCR 文本推断金额、分类、账户、备注
- 语音识别先转文本，再按规则 / AI 回填

原因：

- 降低直接多模态接入复杂度
- 保持账单字段映射可控

### 决策 5：同步能力选 WebDAV / NAS

- 当前同步目标不是项目私有云，而是用户自己的 NAS / WebDAV

原因：

- 对个人用户成本低
- 不需要自建服务
- 用户可掌握数据所有权

### 决策 6：分类体系以“固定大类 + 可追加小类”为主

- 代码里内置了较完整的大类和小类
- 自定义分类是附加到固定组下，而不是完全自由树形结构

原因：

- 统计更稳定
- AI / OCR 分类映射更容易做

代价：

- 灵活度有限
- 分类管理页面也明确提示当前版本不开放完整编辑

### 决策 7：本地备份只留 1 份，NAS 留 9 份

- 最近一次调整后：
  - 本地只保留最新 1 份备份
  - NAS 仍保留最新 9 份
  - WebDAV 同步过程中的本地中间备份会自动删掉

原因：

- Android 本地“用户数据”曾因备份累积明显膨胀
- 用户希望节省手机存储
- NAS 端仍保留更多历史作为兜底

### 决策 8：暂不重构单文件，优先继续交付功能

- 尽管 `lib/main.dart` 很大，但最近几次需求仍是在原结构上快速迭代

原因：

- 当前目标是功能交付速度
- 全量拆分重构风险高

代价：

- 新人 / 新 AI 接手理解成本高

## 4. Codex 开发时的对话要点总结（关键需求变更）

以下是最近一轮开发中，已经确认并落地的关键需求 / 事实，接手者应默认成立。

### 需求变更 1：高级筛选增加“取消全选”

- 页面：高级筛选
- 范围：
  - 分类
  - 账户
- 规则：
  - 都要增加“取消全选”按钮
  - 点击后分别一键取消所有分类 / 所有账户
  - 当内部选项没有全选时，不显示该按钮

### 需求变更 2：本地备份从“多份保留”调整为“只保留 1 份”

- 用户明确表示：手机本地没有必要保留 9 份备份，1 份即可
- 当前代码已经按此调整

### 需求变更 3：NAS 备份策略不改，仍保留 9 份

- 用户确认：
  - 本地保留 1 份即可
  - NAS 继续保留过去的 9 份历史

### 需求变更 4：排查并清理 Android 用户数据膨胀

- 已排查出历史问题：
  - 本地 `ledger_backup_*.json` 大量累积
- 已执行处理：
  - 清理设备上的冗余本地备份
  - 清理 cache 中的分享 / 选图残留
- 结论：
  - 用户数据下降明显
  - 仍保留的约百兆数据不全是垃圾，debug 运行资产也会占空间

### 需求变更 5：Android 安装流程的经验结论

- `flutter build apk --debug` 在这个项目里可能出现“假失败”
- 即使终端说失败，也要继续检查：
  - `android/app/build/outputs/flutter-apk/app-debug.apk`
  - `android/app/build/outputs/apk/debug/app-debug.apk`
- 最近多次安装都依赖这个经验完成

### 需求变更 6：修复 APP 启动时背景全黑闪一下

- 问题：打开 APP 或切换 Tab 时，透明容器叠加导致底层显示黑色，约半秒后背景图加载完成才恢复正常
- 根因：Scaffold 和所有页面容器都是透明背景，完全依赖 `bg.jpg` 一张图打底，图片解码延迟时透出黑色
- 修复方案（已落地）：
  - Android `NormalTheme` 窗口背景色从 `@android:color/white` 改为 `#FFF8FAF6`（与 APP 主题色一致）
  - Flutter `body` Stack 中在背景图**下方**增加一层 `Container(color: Color(0xFFF8FAF6))` 作为后备色
  - 页面容器保持透明，让背景图正常显示在最上层
- 注意：不要把页面容器改成不透明，否则会把背景图完全挡住

### 需求变更 7：添加自动更新功能

- APP 启动时自动检测 GitHub Releases 最新版本
- 有新版本弹窗提示，支持下载安装
- 设置页新增”检查更新”卡片，支持手动检查
- GitHub 仓库：`AKABOOZ/booz-ledger`（Public）
- APK 安装使用 Android 原生 Intent + FileProvider

### 需求变更 8：优化语音记账别名匹配

- 新增大量支出分类别名（快递→邮寄费、房贷→房租、deepseek→AI使用费 等）
- 别名表在 `lib/services/ledger_text_parser.dart` 的 `_matchExpenseCategory` 方法中

### 需求变更 9：修复统计页切换动画问题

- 问题：每次切换到统计页，排行榜卡片会有展开动画
- 原因：页面先用默认月份渲染，再异步恢复保存的月份，数据量变化触发动画
- 解决方案：
  - APP 启动时预加载统计页月份设置（`StatisticsPagePrefs`）
  - 统计页直接用内存中的值初始化，无异步加载
  - 展开/收起改用 `AnimationController` + `SizeTransition`，只由按钮触发

### 需求变更 10：新增账户详情页面

- 在账户页面点击某个账户，跳转到账户详情页
- Hero 卡片显示账户名称、当前余额、最近30天余额变化折线图
- 折线图支持滑动查看不同日期的余额数值
- 流水列表按日期分组显示，点击流水可进入编辑页面
- 账户详情页右上角有”编辑”和”删除”按钮

### 需求变更 11：优化金额键盘弹出逻辑

- 进入编辑流水页面时，默认不弹出金额键盘
- 只有点击金额输入框时才弹出键盘
- 新增记一笔页面时，进入后自动弹出键盘

### 需求变更 12：修复支付宝图标圆角

- 支付宝图标的 `borderRadius` 从 `size * 0.3` 改为 `size * 0.38`，与其他图标保持一致

### 需求变更 13：收入/支出类型选择器溢出修复

- 问题：选择收入/支出类型时，底部出现 “OVERFLOWBOTTOM OVERFLOWED BY 1.5 PIXELS” 错误
- 原因：GridView 的 `childAspectRatio: 1` 导致格子高度不足以容纳图标+文字
- 解决方案：使用 `ClipRect` 裁剪溢出内容，并微调网格高度计算

### 需求变更 14：折线图优化

- 纵轴数字标签移到折线图左侧外面，用动态宽度计算间距
- 纵轴添加5条虚线参考线，每条线左侧显示对应数据
- 去掉折线图上的小圆点，只保留手指滑动时的指示器
- 纵坐标数值去掉 ¥ 字符，超过1万显示”X.X万”，超过1千显示”X.XK”

### 需求变更 15：折线图余额计算修复

- 问题：折线图最后一天的余额与实际账户余额不一致
- 原因：计算时先记录余额再加当天交易，导致最后一天显示交易前的余额
- 修复方案：改为先加当天交易再记录余额

### 需求变更 16：图片识别规则补充

- 新增规则：出现”上海银行信用购”字样时，分类应选”花呗”，类型为 expense
- 该规则添加在 AI 图片识别的 system prompt 中

### 需求变更 17：记一笔键盘加减计算小数点修复

- 问题：输入 `123.56+135.68` 时，第二个小数点无法输入，只能输入 `123.56+13568`
- 原因：小数点逻辑只检查整个表达式是否有小数点，没有按数字段检查
- 修复方案：改为检查当前数字段（最后一个 `+` 或 `-` 之后的部分）是否已有小数点

### 需求变更 18：折线图纵坐标单位统一

- 账户详情页折线图纵坐标从"万"改为"W"（如 1.5W、4.1W）

### 需求变更 19：账户管理页面新增总资产折线图

- 在"当前资产"卡片底部新增近30天总资产变化折线图
- 资产隐藏时，折线图纵坐标和滑动指示器都显示 `****`
- 去掉了"xxx个账户"的副标题

### 对后续 AI 的提醒

- 不要把根目录 React/Vite 工程当成主线
- 不要轻易回退备份清理逻辑
- 修改同步、导入导出、账户余额联动时，要一起验证数据一致性
- 在没有充分预算前，不要贸然做”大拆 main.dart”式重构
- 本项目已初始化 git 仓库，日常修改后应 `git commit`
- GitHub CLI (`gh`) 已安装，可用于自动发布版本到 GitHub Release
- 发版命令：`gh release create v<VERSION> --repo AKABOOZ/booz-ledger --title “波哥记账 v<VERSION>” --notes “更新内容” <APK路径>`
- Flutter debug 模式构建比 release 快很多（约 12s vs 60s），日常迭代测试优先用 debug
